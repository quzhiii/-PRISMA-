# PRISMA Workbench 商业化路径草案

Last updated: 2026-04-28

## 1. 为什么当前 V2.2 继续免费是合理的

当前最重要的资产是信任、传播和方法学可信度。PRISMA Workbench 面向的用户包括研究生、医生、课题组和方法学研究者，他们会优先关心工具是否可靠、数据是否留在本地、输出是否能复核，而不是高级商业功能。

因此，当前不建议把核心本地能力锁到付费墙后面。免费本地版应继续覆盖：

- 单人项目
- 本地导入、去重、筛选、全文复核
- 基础质量评价入口
- PRISMA 2020 图和基础结果表导出
- V2.2 基础审计包导出

## 2. 不建议收费的能力

| 能力 | 原因 |
|---|---|
| 本地导入和字段识别 | 属于工具基本可用性 |
| 保守去重 | 是项目可信度基础 |
| 规则筛选和人工复核 | 是系统综述工作流基本盘 |
| 基础 PRISMA 图导出 | 用户对项目的核心预期 |
| 基础审计包导出 | 支撑 audit-ready 定位，不应变成早期门槛 |
| No-AI 模式 | 必须完整保留，避免用户被迫进入 AI 或云端 |

## 3. 可以收费的能力

更适合付费的能力是“厚层能力”，即协作、治理、模板、AI 成本和机构服务。

| 能力 | 付费理由 |
|---|---|
| 多人协作和项目空间 | 有服务器、同步、权限和支持成本 |
| 高级审计报告 | 机构审计、方法学附录、批量项目汇总有更高价值 |
| 高级质量评价模板 | 需要维护专业模板和导出格式 |
| AI prioritization / suggestion | 有模型成本、提示词维护和合规说明成本 |
| 团队模板包 | 面向课题组、科室、图书馆和方法学平台 |
| 私有部署 | 实施、培训、升级和数据安全责任更重 |
| 中文源增强清洗包 | 可针对真实机构导出格式做定制适配 |

## 4. Free / Pro / Team / Institution 分层

| 层级 | 适合对象 | 能力边界 |
|---|---|---|
| Free Local | 个人研究者、研究生 | 本地单人工作流、基础质量评价、基础审计包、PRISMA 导出 |
| Pro Local | 高频个人用户、小课题负责人 | 高级质量模板、高级审计摘要、AI usage report、批量导出整理 |
| Team | 课题组、科室、小型研究团队 | 多人项目、角色分工、冲突解决、团队模板、项目共享 |
| Institution | 医院循证中心、高校图书馆、研究所 | 私有部署、机构模板、培训、审计治理、中文源定制 |

当前代码不应实现这些分层，只保留为路线草案。

## 5. 可能的定价逻辑

这些数字只用于后续访谈验证，不应现在写入产品页面。

| 类型 | 逻辑 |
|---|---|
| 个人 Pro | 低价订阅或一次性买断，重点验证 AI / 模板 / 审计增强是否值得付费 |
| Team | 按项目空间、席位或团队工作区收费 |
| Institution | 年度 license、私有部署、培训和维护服务 |
| 定制服务 | 按中文源适配、机构模板、数据迁移和审计报告需求报价 |

## 6. 面向中文研究者的商业化切入点

中文研究者的痛点往往不是“有没有 AI”，而是：

- 多数据库导出格式混乱
- 中文摘要、作者、期刊、基金、机构字段噪音多
- 导入后需要大量手工清洗
- 质量评价和结果表需要反复整理
- 导师、课题组或科室需要能看懂筛选过程

可验证的付费候选：

1. 中文源导入增强包。
2. 质量评价模板包。
3. 投稿导出整理包。
4. AI 使用透明报告和方法学附录。
5. 小团队双人复核和冲突解决。

## 7. 面向机构用户的商业化切入点

机构用户更关心治理、培训和可复用流程。

| 机构类型 | 可能需求 |
|---|---|
| 医院循证中心 | 方法学模板、项目审计、双人复核、培训 |
| 高校图书馆 | 数据库导出教程、教学 demo、批量课程支持 |
| 科研管理部门 | 项目级审计包、方法学附录、统一流程 |
| 研究所方法学平台 | 私有部署、模板维护、团队协作 |

## 8. 风险

| 风险 | 说明 | 控制方式 |
|---|---|---|
| 过早收费伤害传播 | 早期用户尚未建立信任 | 核心本地版继续免费 |
| 商业层过重拖慢开发 | 后端、账号、权限会显著增加复杂度 | 先做文档和访谈验证 |
| AI 成本和责任不清 | 用户可能误解 AI 能保证结论 | 强制 conservative AI 和 audit report |
| 中文源定制碎片化 | 每个机构导出都可能不同 | 先沉淀字段映射和 fixture |
| 与商业平台正面竞争 | Rayyan/Covidence/DistillerSR 组织采购强 | 避免早期做重企业平台 |

## 9. 下一步验证方式

1. 找 3-5 个真实系统综述用户试用 V2.2 audit-ready workflow。
2. 记录他们愿意为哪类能力付费：模板、AI、协作、审计、中文源、部署。
3. 对比个人、课题组、医院/高校机构三类用户的预算逻辑。
4. 不写支付代码，只用访谈和手工交付验证需求。
5. 在 V2.3/V2.4 后再决定是否建立 Pro/Team 产品层。

## 10. 当前版本禁区

当前版本不要加入：

- 登录系统
- 支付系统
- license check
- 默认云端账号
- 默认云 AI API
- 强制上传项目数据
- 通过功能锁限制基础本地导出

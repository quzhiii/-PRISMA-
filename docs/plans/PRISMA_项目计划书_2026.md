# PRISMA 项目计划书（2026 版）

## 一、项目定位

### 1.1 项目一句话定位

**PRISMA = 面向系统综述、Meta 分析与证据整合的本地优先（local-first）、可审计（audit-ready）、保守型 AI 增强文献筛选工作台。**

### 1.2 当前产品本质

PRISMA 现在不应再被定义为“PRISMA 流程图生成工具”，而应被定义为：

- 一个覆盖导入、去重、规则筛选、人工复核、质量评价、导出的端到端工作流产品
- 一个强调“数据默认不离开设备”的浏览器本地优先工具
- 一个把“误删风险控制、人工责任保留、过程留痕”放在效率之前的筛选系统
- 一个对中文源、多源脏数据、研究设计差异更敏感的 evidence screening workstation

### 1.3 核心用户

第一层核心用户：

- 做系统综述、范围综述、快速综述、Meta 分析的研究生与青年研究者
- 医学、公卫、护理、药学、卫生政策、教育、心理、社科等需要规范筛选流程的人群

第二层扩展用户：

- 高校图书馆证据综述支持人员
- 医院科研管理部门、循证中心、方法学团队
- 需要在本地处理数据、对隐私和可审计性有要求的小团队

第三层未来用户：

- 机构级 evidence synthesis 团队
- 面向中文数据库和双语研究场景的学校/研究机构

## 二、为什么这个项目有机会

### 2.1 市场不是空白，而是碎片化

当前 evidence synthesis 工具非常多，但公开市场呈现出明显分层：

- 商业平台强在协作、组织采购、端到端完整性
- 开源工具强在某个单点，如 active learning、Git 协作、全文抽取、RoB 可视化
- 很少有工具同时把 local-first、audit-ready、quality appraisal、Chinese-source compatibility、conservative AI 做深

这意味着 PRISMA 的机会不在于“再做一个 AI 筛选工具”，而在于做出**组合差异化**。

### 2.2 你当前已经踩中的稀缺点

PRISMA 现有版本已经具备以下潜在护城河：

- 浏览器本地运行，降低隐私顾虑与部署门槛
- 双层去重，强调“硬重复自动移除 + 疑似重复人工复核”
- 质量评价 / 证据等级已进入主流程，不再是后置补丁
- 已开始处理中文源真实脏数据（如 CNKI RDF 摘要噪音）
- 产品哲学是保守优先，而非激进自动化

### 2.3 未来需求方向并不支持“无脑自动化”

未来 1–2 年，AI 在 evidence synthesis 的方向不会只是“更快筛文献”，而是：

- AI 使用过程必须更可报告
- 人机分工边界必须更清楚
- 过程日志、理由留痕、可回放能力会越来越重要
- 高 recall、低误删、可解释工作流比“黑盒自动筛完”更有长期价值

## 三、战略定位建议

### 3.1 不建议的定位

现阶段不建议把 PRISMA 对外定义为：

- “PRISMA 流程图工具”
- “通用 AI 文献综述平台”
- “Scientific Agent Governance Stack” 的直接产品入口

原因：

- 第一种过窄，无法承接你已做出的工作流厚度
- 第二种过宽，会直接进入和 Rayyan / Covidence / ReviewAid / 各类 AI 平台的正面竞争
- 第三种叙事适合作为你的项目矩阵上位品牌，不适合作为 PRISMA 单项目的一号标题

### 3.2 建议采用的定位

建议对外使用：

> **PRISMA 是一个本地优先、可审计、保守型 AI 增强的文献筛选工作台，服务于严谨综述与证据整合场景。**

### 3.3 与你其他项目的关系

建议把关系拆成两层：

项目层：

- PRISMA：Evidence Layer（证据筛选与审计）
- runtime-governance：Runtime Layer（分析/编码过程治理）
- thesis-skills：Artifact Layer（论文产物与提交前检查）

品牌层：

- 可以逐步汇总为你更上位的“研究治理 / Scientific Agent Governance” 叙事
- 但 PRISMA 当前仍应以 evidence screening workstation 的单点价值为主

## 四、未来演进方向与具体路径

### 路径 A：先做深本地优先的筛选工作台（近期主线）

目标：把 PRISMA 做成一个真正可被持续使用的 screening workstation，而不是演示型工具。

具体方向：

- 导入稳定性继续增强
- 去重解释能力增强
- 单/双人复核流程更稳
- 质量评价模板化
- 导出更接近投稿/附录实际需要
- 明确支持中文源与双语项目

该路径的价值：

- 与现有能力连续性最好
- 用户价值最清晰
- 容易建立口碑
- 最适合作为免费版/开源版长期存在

### 路径 B：在 A 基础上增加保守型 AI 辅助层（中期主线）

目标：让 AI 成为“建议层、优先级层、不确定样本提示层”，而不是替代判断层。

具体方向：

- AI 仅做优先级排序
- AI 给出纳入/排除理由建议，但不默认直接排除
- AI 标记边界样本、冲突样本、可疑摘要
- AI 协助研究设计识别与质量评价模板推荐
- 自动生成 AI 使用记录与方法附录说明

该路径的价值：

- 与未来报告规范趋势一致
- 保持你的“保守优先”品牌
- 能形成差异化，而不是卷通用聊天能力

### 路径 C：向机构级协作与审计层扩展（远期商业主线）

目标：从个人工具升级到团队/机构级解决方案。

具体方向：

- 项目级审计账本
- 协作权限与角色管理
- reviewer calibration / IRR 追踪
- 机构模板与方法学规范包
- 组织级 review dashboard
- SSO / 管理后台 / 合规部署

该路径的价值：

- 是未来最可能承接收费和机构采购的部分
- 能与免费本地版形成清晰分层

## 五、Roadmap（建议版）

## Phase 0：产品定义收束（0–1 个月）

目标：先把“你到底在卖什么”说清楚。

交付：

- README 重写
- landing page 信息架构重写
- 产品一句话定位统一
- 功能边界说明（本地优先 / 可审计 / AI 辅助而非自动裁决）
- 中英文对照文案统一

验收标准：

- 新用户在 30 秒内能理解 PRISMA 不是“单纯画图工具”
- 能一眼看出与 Rayyan / Covidence / ASReview 的差异

## Phase 1：把 V2.1 打磨成稳定可用版（1–3 个月）

目标：把基础工作台做扎实。

交付：

- 更完整的导入兼容矩阵
- 去重结果解释页
- 筛选原因结构化标签
- 质量评价模板化（至少覆盖高频研究类型）
- 导出结果增强（明细表、附录表、筛选摘要）
- 中文源字段清洗与规范化增强

验收标准：

- 单个综述项目可以在工具内完成从导入到导出的主流程
- 用户无需借助多个额外工具来补质量评价和结果整理

## Phase 2：审计层上线（2–5 个月）

目标：把“audit-ready”变成显式能力，而不是隐含哲学。

交付：

- screening ledger（逐条记录的过程账本）
- decision history（纳入/排除/修改原因变更记录）
- AI usage report（何时调用 AI、输入输出边界、责任提示）
- methods appendix generator（生成可写入方法部分/附录的过程摘要）

验收标准：

- 用户可以直接导出一套审稿/附录友好的流程说明
- 项目内每条记录的处理路径可追溯

## Phase 3：保守型 AI 模块（4–8 个月）

目标：引入 AI，但不破坏方法学可信度。

交付：

- AI prioritization mode
- uncertain bucket（不确定样本桶）
- include/exclude rationale suggestion
- research design suggestion
- quality appraisal starter suggestions
- 多模型/本地模型实验接口

验收标准：

- AI 默认不静默排除文献
- AI 的每次介入都有日志与可关闭开关

## Phase 4：轻协作与付费试点（6–12 个月）

目标：验证收费可能性。

交付：

- 共享项目
- reviewer roles
- 冲突解决工作流
- reviewer calibration
- 私有项目空间
- AI 额度管理
- 机构试点方案

验收标准：

- 出现真实付费意愿，而不仅是 GitHub star
- 至少有一类用户愿意为“协作 / AI / 审计 / 部署”付费

## 六、功能优先级（P0 / P1 / P2）

### P0：必须先做

1. **产品定位重写**
   - README / 首页标题 / 功能框架统一

2. **审计账本基础结构**
   - 逐条记录导入、去重、筛选、复核、导出动作

3. **质量评价模板化**
   - 从“有入口”变成“可直接用”

4. **中文源兼容加强**
   - CNKI / 双语摘要 / 字段噪音 / 编码异常 / 期刊字段不规范处理

5. **导出增强**
   - PRISMA 图之外，增加附录、过程摘要、结构化结果表

### P1：重要但可第二批

1. **AI 排序与提示层**
2. **不确定样本桶**
3. **双人复核校准 / IRR 辅助**
4. **研究设计识别增强**
5. **质量评价图形输出**

### P2：可以后置

1. **全自动全文抽取**
2. **过重的云端协作系统**
3. **一开始就做复杂机构后台**
4. **泛化成所有科研场景的平台**

原因：

- 这些功能成本高、边界复杂、容易把产品带偏
- 现阶段更应该先把“严谨筛选工作台”这个单点打透

## 七、商业化路径

### 7.1 是否应该收费

可以探索收费，但**不建议直接把当前 V2.1 的核心本地能力整体锁付费**。

更合理的原则是：

- 免费版保留单人、本地、基础筛选主流程
- 收费版承接协作、AI、审计增强、机构治理、部署服务

原因：

- 你的现阶段品牌资产来自“本地优先、开放、可信、低门槛”
- 如果过早把核心能力锁死，会削弱传播与口碑
- 商业平台的强项是组织采购，你更适合先用免费版建立用户信任，再把付费点放在更厚的层上

### 7.2 建议的收费分层

#### 方案 A：Open-core（最推荐）

免费开源层：

- 单人项目
- 本地导入 / 去重 / 筛选 / 基础导出
- 基础 PRISMA 图
- 基础质量评价模板

付费增强层：

- 双人/多人协作
- 审计账本高级导出
- AI prioritization 与建议层
- 私有项目空间
- 团队模板包
- 机构管理与权限
- 私有部署 / 本地部署支持

适用原因：

- 最符合你当前社区阶段
- 同时兼顾 GitHub 开源传播与后续商业化

#### 方案 B：Hosted SaaS + Local 免费版

免费版：

- 本地浏览器版继续免费

付费版：

- 云端同步
- 团队协作
- AI 调用
- 项目空间与备份
- 组织管理

适用原因：

- 容易解释
- 与 local-first 形成清晰区分

风险：

- 你需要维护后端、账号体系、权限、数据安全
- 这会显著增加产品复杂度

#### 方案 C：机构服务 / 私有部署优先

目标客户：

- 高校图书馆
- 医院循证中心
- 研究所方法学平台
- 中文数据库使用频繁的本地机构

收费内容：

- 私有部署
- 机构模板包
- 中文源适配增强
- 培训与实施支持
- 组织级审计与导出

适用原因：

- 更符合“可信、可审计、可治理”的定位
- 可能比做散户订阅更接近你的长期优势

### 7.3 商业化顺序建议

建议按以下顺序推进：

1. **先不动核心免费版**
2. **先做出清晰的付费层候选功能**
3. **先拿 3–5 个真实用户访谈或试用反馈**
4. **先验证谁愿意付费：个人、课题组、图书馆、医院方法学团队**
5. **最后再决定主商业模式是订阅还是机构服务**

### 7.4 初始定价方向（仅作探索区间）

个人版：

- 低价订阅，适合作为 AI / 审计增强包
- 不建议一开始定太高

团队版：

- 按项目空间或按席位收费
- 可提供协作、审计、模板、共享空间

机构版：

- 按年度 license / 部署服务 / 培训支持收费
- 这是更可能形成收入体量的方向

## 八、命名是否需要改

### 8.1 当前名称的问题

`PRISMA` 有几个现实问题：

1. **过于通用**：PRISMA 本身是方法学通用术语，不利于搜索与品牌占位
2. **误导性偏窄**：容易让人以为你只做流程图，而不是完整工作台
3. **与官方标准语义过近**：虽然有借势好处，但长期不利于形成独立品牌
4. **GitHub 仓库名 `-PRISMA-` 可读性一般**：检索与传播都不够理想

### 8.2 现在要不要立刻改名

**建议：可以改，但不要急于把“PRISMA”完全丢掉。**

更稳妥的做法是分两层：

- 产品品牌名：换成一个更像产品的名字
- 副标题：保留 “PRISMA workflow / evidence screening / systematic review” 这些高识别关键词

例如：

> 新名字 + subtitle: local-first PRISMA screening workstation

### 8.3 命名策略建议

建议优先满足四个条件：

- 不只是流程图含义
- 更像产品名而不是方法学名词
- 能承接未来协作 / 审计 / AI 模块
- 中英文传播都不太别扭

### 8.4 命名方向示例

#### 方向 A：保守过渡型（推荐）

- **Prisma Workbench**
- **Prisma Studio**
- **Prisma Desk**
- **Prisma FlowLab**

优点：

- 延续现有认知
- 不会让老用户迷失
- 便于 README/SEO 迁移

缺点：

- 仍然绑定 PRISMA 语义，独立性有限

#### 方向 B：独立产品型

- **Evidesk**
- **Screenory**
- **Evidra**
- **ReviewDesk**
- **Evidence Studio**

优点：

- 更利于长期品牌化
- 不局限于流程图语义

缺点：

- 初期需要重新建立认知

#### 方向 C：中文场景 / 双语场景导向

- **证筛台 / 证筛工作台**
- **循证筛选台**
- **综述工作台**

优点：

- 中文语义清楚
- 适合国内传播

缺点：

- 国际化品牌感相对弱

### 8.5 我的建议

现阶段更推荐：

- **对外品牌先过渡，不要硬切**
- 先用 `PRISMA Workbench` 或 `PRISMA Studio` 这类名字过渡一版
- 等协作 / 审计 / AI 模块更成熟后，再评估是否彻底切换为独立品牌

## 九、与 Kimi 那份大叙事的关系：怎么处理更合适

Kimi 那份分析里，“Scientific Agent Governance / 研究治理栈”的判断有启发，但更适合作为**你的项目矩阵上位叙事**，而不是 PRISMA 当前的一号产品定位。

更合理的处理方式是：

- PRISMA 先讲清楚自己是 evidence screening workstation
- runtime-governance 讲 runtime governance
- thesis-skills 讲 artifact / submission governance
- 然后在你的个人主页、总览仓库或方法论文章中，再把它们汇总为统一治理框架

这样做的好处：

- 单项目入口不会被抽象概念稀释
- 项目之间仍然可以形成闭环叙事
- 市场理解成本更低

## 十二、一个可直接采用的新版定位文案

### 中文版

PRISMA 是一个面向系统综述、Meta 分析与证据整合场景的本地优先文献筛选工作台。它覆盖导入、去重、规则筛选、人工复核、质量评价与结果导出，并强调可审计、可解释和保守型 AI 辅助，而不是黑盒自动裁决。

### 英文版

PRISMA is a local-first evidence screening workbench for systematic reviews, meta-analyses, and evidence synthesis. It covers import, deduplication, screening, human review, quality appraisal, and export, with an emphasis on auditability, interpretability, and conservative AI assistance rather than black-box automation.

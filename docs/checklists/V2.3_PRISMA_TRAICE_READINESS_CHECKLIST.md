# V2.3 PRISMA-trAIce Readiness Checklist

Status: release-readiness checkpoint for the local V2.3 mock/audit layer.

## Scope

- Keep AI mode `off` by default.
- Keep assistive AI suggestions local and mock-only.
- Do not connect a real AI provider in V2.3.
- Keep the provider abstraction request-draft-only: no automatic cloud dispatch and no API key export.
- Treat every AI suggestion as advisory until a human accept/edit action creates a `ScreeningDecision`.
- Keep rejected suggestions outside PRISMA counts.

## Frozen Export Surface

The V2.3 audit package should continue to include:

| File | Readiness expectation |
|---|---|
| `project_manifest.json` | Project metadata and export manifest remain stable. |
| `events.jsonl` | Audit event stream remains append-only and replayable. |
| `screening_decisions.csv` | Final screening decisions come from human actions. |
| `exclusion_reasons.csv` | Exclusion taxonomy and counts remain exportable. |
| `prisma_counts.json` | PRISMA counts are recalculated from human-confirmed decisions. |
| `audit_summary.md` | Human-readable audit summary remains available. |
| `ai_usage_registry.json` | AI mode, provider boundary, allowed stages, and confirmation requirements are exported. |
| `ai_suggestions.jsonl` | Suggestions include hashes, human actions, linked decisions, review trace fields, and PRISMA count boundary. |
| `PRISMA_TRAICE_REPORT.md` | No-AI and assistive-AI transparency statements are exportable. |

## Required Behavior

- No-AI mode exports a clear No-AI statement in `PRISMA_TRAICE_REPORT.md`.
- Assistive mode registers the mock AI usage boundary in `ai_usage_registry.json`.
- Provider metadata is redacted before export; API key-like fields must not appear in audit files.
- Provider request drafts record prompt/input hashes and `not_dispatched` status until a later explicit API integration slice.
- Provider configuration UI remains boundary-only: it records provider type, model, endpoint origin, data boundary, and allowed stages without collecting API keys or dispatching requests.
- Mock suggestions are deduplicated by record, stage, model, and prompt hash.
- Accepting an AI suggestion creates a human-confirmed `ScreeningDecision`.
- Editing an AI suggestion creates a human-confirmed `ScreeningDecision` using the explicit human-selected decision and exclusion reason.
- Rejecting an AI suggestion only updates the suggestion review trace and does not affect final decisions or PRISMA counts.
- `ai_suggestions.jsonl` exports `reviewed_at`, human edit fields, review notes, `linked_decision_id`, and `prisma_count_boundary`.
- Audit events continue to record `ai_suggestion_generated` and `ai_suggestion_reviewed` events.
